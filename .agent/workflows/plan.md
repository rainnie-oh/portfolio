# Plan (计划输出模式)

**触发时机**：我说“Plan”、“拆一下”，或者当我提出大规模的新增功能/改写意图时。

在写具体代码前，你必须严格按照下面 5 项结构输出你的分析与计划（每项 1-2 行即可，不要长篇大论）：

1. **Goal**：这次明确想达成什么？
2. **Non-goals**：这次边界在哪，明确不做什么？
3. **Acceptance**：≤5 条验收标准（必须是可独立验证的结果）。
4. **Open decisions**：需要我（用户）拍板的点（≤3个；如果在你的架构判断下目前没有，就写 None）。
5. **Risk + Rollback**：主要风险点是什么？如果不顺怎么回退？

**同时在计划后简短补充“合理架构”判断：**
👉 说明新增内容放哪里、是否会导致组件耦合、组件是“进 DS 加复用”还是“页面私有”。
